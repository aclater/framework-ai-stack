# Grounding specification

## Principle: corpus-preferring with transparent fallback

The RAG proxy implements a grounding strategy where retrieved documents are the primary source of truth, but the model may draw on general knowledge when the corpus is silent. The key requirement is **transparency** — the caller always knows whether the answer came from the corpus or general knowledge.

This is intentionally not a strict "only answer from documents" policy. A strict policy would refuse to answer any question not covered by the corpus, which is unhelpful for users who expect a knowledgeable assistant. Instead, the model answers from general knowledge when needed but explicitly marks those answers so callers can assess confidence.

## System prompt

The exact system prompt sent to the model:

```
You are a knowledgeable assistant with access to a curated
document corpus. Use the following rules when answering:

1. If the retrieved documents contain relevant information,
   use them as your primary source. Cite every claim drawn
   from the documents using [doc_id:chunk_id].

2. If the retrieved documents are silent or insufficient,
   you may draw on your general knowledge to answer. When
   doing so, you must prefix your response with:
   "⚠️ Not in corpus:" to indicate the answer is not
   sourced from the provided documents.

3. If the retrieved documents partially answer the question,
   answer the covered portion from documents with citations,
   then address the remainder from general knowledge prefixed
   with "⚠️ Not in corpus:" for that portion only.

4. Never present general knowledge as if it came from the
   retrieved documents. Never fabricate citations.

5. If you are uncertain whether your general knowledge is
   accurate, say so explicitly rather than stating it
   with false confidence.
```

The prompt hash is checked in tests to ensure it doesn't drift without updating the test assertion.

## Citation format

Documents are presented to the model with citation labels:

```
[3ea96246-a808-5fbb-8d34-d8153eb5002f:13] (Source: gdrive://NATO_PQC_report.docx)
The Alliance published its Quantum Technologies Strategy in January 2024...
```

The model cites claims using `[doc_id:chunk_id]` references. These are parsed from the response by regex (`[a-f0-9-]+:\d+`), not generated by the model as structured output.

## Grounding modes

| Mode | Meaning | When it occurs |
|------|---------|---------------|
| `corpus` | Entire answer sourced from retrieved documents | Model cites documents, no ⚠️ prefix |
| `general` | Entire answer from general knowledge | No valid citations, or corpus_coverage is "none" |
| `mixed` | Part from corpus (with citations), part from general knowledge | Both citations and ⚠️ prefix present |

## Response metadata

Every non-streaming response includes a `rag_metadata` block:

```json
{
  "rag_metadata": {
    "grounding": "corpus",
    "cited_chunks": ["3ea96246:13", "3ea96246:12"],
    "corpus_coverage": "full"
  }
}
```

- **grounding** — `corpus`, `general`, or `mixed`
- **cited_chunks** — list of validated `doc_id:chunk_id` strings
- **corpus_coverage** — `full` (chunks retrieved), `partial`, or `none` (empty retrieval)

This metadata is populated by **parsing the response**, not by asking the LLM to generate it. The grounding classification is deterministic based on the presence of citations and the ⚠️ marker.

## Citation validation

After the model responds, every cited `[doc_id:chunk_id]` is validated:

1. **Was it in the retrieved set?** The citation must reference a chunk that was actually returned by Qdrant and passed to the model. Citations to chunks the model didn't see are hallucinated.

2. **Does it exist in the document store?** The `(doc_id, chunk_id)` pair must exist in the Postgres chunks table.

If a citation fails validation:
- It is **stripped** from the response text
- The rest of the response is preserved (it may contain legitimate general knowledge)
- An error is logged with `query_hash`, the invalid citation, and the failure reason
- The metadata shows `citation_validation: "stripped"`

Invalid citations are never silently passed through, and valid response content is never discarded because of a bad citation.

## Empty retrieval behavior

When no Qdrant results pass the reranker threshold (or the collection is empty):

1. `corpus_coverage` is set to `"none"`
2. The system message includes: "No relevant documents were retrieved for this query."
3. The model proceeds and answers from general knowledge with the ⚠️ prefix
4. The retrieval miss is logged at info level with the query hash and score distribution

This is **not an error condition**. Empty retrievals are valuable signal for corpus gap analysis — they tell you what questions your document corpus can't answer.

## Reranker behavior

The reranker (cross-encoder/ms-marco-MiniLM-L-6-v2) scores all Qdrant candidates and returns the top-N:

- `RAG_TOP_K` (default 20) — candidates sent to Qdrant
- `RERANKER_TOP_N` (default 5) — results after reranking
- `RERANKER_MIN_SCORE` (default -999) — soft threshold, not a hard stop

Below-threshold results don't block the query. The model still gets called, just with fewer or no context chunks.

## Thinking budget

The model runs with thinking enabled and a token budget:

- `THINKING_BUDGET` (default 1024) — controls `max_completion_tokens` when the caller doesn't set their own
- Actual limit: `THINKING_BUDGET + 2048` tokens to allow both reasoning and response
- Allows the model to reason across retrieved chunks and general knowledge without unbounded latency

## Audit log schema

Every query produces a structured audit log entry (JSON, info level on the `rag-proxy.audit` logger):

```json
{
  "timestamp": "2026-04-02T21:05:22.890Z",
  "query_hash": "sha256 of raw query text",
  "retrieved_chunks": [
    {"doc_id": "...", "chunk_id": 0, "reranker_score": 0.95}
  ],
  "chunks_passed_threshold": 5,
  "grounding": "corpus",
  "corpus_coverage": "full",
  "cited_chunks": [
    {"doc_id": "...", "chunk_id": 0}
  ],
  "citation_validation": "pass",
  "response_type": "answered"
}
```

**Privacy constraint:** The audit log never contains query text, response text, or document content. Only hashes, IDs, scores, and classification metadata. This allows monitoring and corpus gap analysis without exposing sensitive content.

## Streaming limitation

Streaming responses (`stream: true`) cannot be post-processed for citation validation because the full response text isn't available until the stream completes. Streaming requests pass through without grounding metadata or citation validation. Use non-streaming requests when grounding metadata is needed.
